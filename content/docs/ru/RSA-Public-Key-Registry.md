# Реестр открытых ключей RSA

GoyGram жестко кодирует все 8 открытых ключей RSA Telegram, используемых для обмена ключами DH. Это ключи, которые подписывают `p_q_inner_data` во время генерации ключа аутентификации.

## Карта ключей


```python
TELEGRAM_RSA_KEYS: dict[int, int] = {
    # Fingerprint → 2048-bit RSA modulus
    847625836280919973:  int("2208194653103..."),
    1562291298945373506: int("2397875855310..."),
    -5859577972006586033: int("2271864697902..."),
    6491968696586960280: int("2403776680080..."),
    -4344800451088585951: int("2440344664914..."),
    -7306692244673891685: int("2508140781041..."),
    -5738946642031285640: int("2234733764462..."),
    8205599988028290019: int("2457345520795..."),
}
```


Каждый ключ представляет собой 2048-битный модуль RSA (617 десятичных цифр). Публичный показатель всегда равен 65537.

## Выбор клавиши

Во время обмена DH сервер возвращает список отпечатков пальцев в `resPQ`. GoyGram выбирает первый подходящий:


```python
fp = next((x for x in fps if x in TELEGRAM_RSA_KEYS), None)
if fp is None:
    raise RuntimeError(f"no known Telegram RSA key fingerprint in resPQ: {fps!r}")
```


Если ни одно совпадение не соответствует (например, Telegram поменял все ключи), аутентификация не удалась. Такого никогда не случалось в производстве — ключи RSA Telegram чрезвычайно долговечны.

## Схема заполнения RSA (`rsa_pad_encrypt`)

Точный алгоритм из `tl_core.py`:


```python
def rsa_pad_encrypt(data: bytes, n: int, e: int) -> bytes:
    d = sha1(data).digest() + data           # SHA-1 hash (20 bytes) + raw data
    d += secrets.token_bytes(255 - len(d))    # random padding to exactly 255 bytes
    return pow(int.from_bytes(d, 'big'), e, n).to_bytes(256, 'big')
```


**Шаг за шагом:**

1. **Добавление хеша**: к данным добавляется `SHA-1(data)` (20 байт). Сервер проверяет этот хэш, чтобы обнаружить подделку.
2. **Случайное заполнение**: объединенные байты дополняются криптографически случайными байтами до общего объема ровно 255 байт. 2048-битный модуль RSA может зашифровать не более 255 байт.
3. **Шифрование RSA**: `m^65537 mod n` — стандартный RSA с общедоступным показателем 65537. Результат — ровно 256 байт (2048 бит).

**Почему 65537?** Это стандартная общедоступная экспонента RSA — достаточно большая, чтобы избежать атак с малой экспонентой, и достаточно маленькая для быстрого шифрования (17 умножений методом квадрата и умножения).

## Как устроен `p_q_inner_data`

Перед шифрованием RSA создаются внутренние данные, закодированные в TL:


```python
def p_q_inner_data(self, *, pq, p, q, nonce, server_nonce, new_nonce):
    return (
        u32(0x83c95aec)           # constructor ID: p_q_inner_data
        + tl_bytes(pq)            # PQ product
        + tl_bytes(p)             # factor P (4 bytes)
        + tl_bytes(q)             # factor Q (4 bytes)
        + nonce                   # client nonce (16 bytes)
        + server_nonce            # server nonce (16 bytes)
        + new_nonce               # client new nonce (32 bytes)
    )
```


Итого: ~100–110 байт данных, закодированных в TL → дополненных SHA-1 (20Б) + случайный (~125Б) → 255 байт → шифрование RSA → 256 байт.

## Шифрование RSA (продолжение)

Затем выбранный ключ используется:


```python
n_mod = TELEGRAM_RSA_KEYS[fp]
e = 65537
inner = self.codec.p_q_inner_data(pq=pq, p=p.to_bytes(4,'big'), q=q.to_bytes(4,'big'),
                                   nonce=nonce, server_nonce=server_nonce,
                                   new_nonce=new_nonce)
enc = rsa_pad_encrypt(inner, n_mod, e)
```


## Почему 8 ключей?

Telegram поддерживает 8 рабочих пар ключей RSA. Это обеспечивает:

1. **Постепенная ротация**: ключи можно менять по одному, не нарушая работу всех клиентов одновременно.
2. **Географическое распространение**. Разные контроллеры домена могут иметь разные отпечатки ключей в `resPQ`.
3. **Резервирование**: если один ключ скомпрометирован, остальные 7 по-прежнему защищают сеансы.

## Модель безопасности

Ключи RSA обеспечивают **аутентификацию установления связи DH**, а не шифрование самого трафика. Модель безопасности:


```
RSA (2048-bit, hardcoded) → protects p_q_inner_data
  │
  └→ DH (2048-bit, ephemeral) → generates auth_key (256 bytes)
        │
        └→ AES-256-IGE (with msg_key derivation) → encrypts all subsequent traffic
```


Даже если все ключи RSA были скомпрометированы, **обмен DH обеспечивает прямую секретность**: ключи RSA шифруют только `p_q_inner_data` во время первоначальной аутентификации. После завершения DH `auth_key` сеанса не зависит от ключей RSA. Компрометация ключей RSA НЕ приводит к расшифровке записанного трафика MTProto задним числом.

`auth_key` (256 байт) является результатом `g^ab mod p` — стандартного общего секрета Диффи-Хеллмана. Знание закрытых ключей RSA позволит вам создать НОВЫЙ ключ аутентификации MITM, но не сможет восстановить ключи аутентификации PAST.

## Обоснование жесткого кодирования

Ключи жестко запрограммированы, а не извлекаются, потому что:
1. Меняются крайне редко (Telegram никогда не менял их в продакшене)
2. Для их динамического получения потребуется доверие к сетевому источнику во время начальной загрузки аутентификации.
3. Сам обмен DH обеспечивает прямую секретность — ключи RSA защищают только начальное рукопожатие.
4. Жесткое кодирование устраняет двусторонние обходы сети и режим сбоя на наиболее критическом этапе установки соединения.

## Обновление ключей

Если Telegram когда-либо меняет ключи:
1. Добавьте новые ключевые записи в `TELEGRAM_RSA_KEYS`.
2. Формат отпечатка пальца представляет собой 64-битное целое число со знаком (некоторые из них являются отрицательными, поскольку в схеме TL они интерпретируются как знаковые).
3. Никаких других изменений кода не требуется — поиск осуществляется автоматически посредством сопоставления отпечатков пальцев.
