# Ядро расширения Rust

Собственная библиотека Rust (`goygram.ext`) — это бьющееся сердце производительности GoyGram. Через него проходит каждый зашифрованный пакет MTProto и каждое хранилище сеансов. Без него фреймворк был бы игрушкой.

## Сборка и распространение

### Система сборки

GoyGram использует **maturin** (инструмент сборки PyO3) для компиляции крейта Rust в модуль расширения Python:


```toml
# pyproject.toml
[build-system]
requires = ["maturin>=1.7,<2.0"]
build-backend = "maturin"

[tool.maturin]
manifest-path = "ext_rust/Cargo.toml"
python-source = "."
module-name = "goygram.ext"
bindings = "pyo3"
```


### Ржавый ящик


```toml
# ext_rust/Cargo.toml
[package]
name = "goygram_ext"
version = "0.5.5"
edition = "2021"

[dependencies]
pyo3 = { version = "0.21.2", features = ["extension-module", "abi3", "abi3-py311"] }
aes = "0.8.4"
aes-gcm = "0.10.3"
cipher = "0.4.4"

[profile.release]
strip = true
lto = true
opt-level = 3
```


Ключевые детали:
- **`abi3-py311`**: расширение предназначено для стабильного ABI Python (3.11+). Один скомпилированный код `.so` работает на Python версий 3.11–3.14+ без перекомпиляции.
- **`strip = true`**: символы отладки удалены из сборок выпуска.
- **`lto = true`**: оптимизация времени соединения для максимальной производительности.
- **`opt-level = 3`**: агрессивная оптимизация.

### Загрузка в Python


```python
# goygram/security.py
try:
    from goygram import ext as _rx
except Exception:
    _rx = None
```


Расширение не является обязательным во время импорта, но становится жестким требованием, когда вы пытаетесь что-либо зашифровать/дешифровать. И `security.py`, и `mtproto.py` проверяют `if _rx is None: raise RuntimeError(...)` перед использованием криптографических операций.

## Экспортированные функции

### AES-256-IGE (шифрование пакетов MTProto)

Режим IGE (Infinite Garble Extension) — это собственный режим блочного шифрования Telegram. Это не стандартный режим — это критический путь для каждого сообщения MTProto.


```
aes_ige_enc(data, key, iv)    → encrypt + auto-pad (PKCS7)
aes_ige_dec(data, key, iv)    → decrypt + auto-unpad
aes_ige_enc_raw(data, key, iv) → encrypt raw (no padding)
aes_ige_dec_raw(data, key, iv) → decrypt raw (no padding)
```


**Ограничения, введенные в Rust:**
- `key` должен содержать ровно 32 байта (AES-256).
- `iv` должен быть ровно 32 байта (IGE использует два 16-байтовых дополнительных IV).
- `data` должен быть кратен 16 байтам (для необработанных вариантов).

**Алгоритм (необработанное шифрование):**

```
x = iv[0:16]   # first half of IV
y = iv[16:32]  # second half of IV
for each 16-byte block in data:
    tmp = block XOR x
    tmp = AES_encrypt(tmp)
    out = tmp XOR y
    x = out
    y = block
```


**Алгоритм (необработанная расшифровка):**

```
x = iv[0:16]
y = iv[16:32]
for each 16-byte block in data:
    tmp = block XOR y
    tmp = AES_decrypt(tmp)
    out = tmp XOR x
    x = block
    y = out
```


Вариант `enc` добавляет заполнение PKCS7 перед шифрованием. Вариант `dec` удаляет заполнение PKCS7 после расшифровки с проверкой правильности всех байтов заполнения.

### AES-256-GCM (шифрование хранилища)

Используется для системы хранилища сеансов:


```
aes_gcm_encrypt(key, nonce, plaintext, aad) → ciphertext+tag
aes_gcm_decrypt(key, nonce, ciphertext, aad) → plaintext
```


**Ограничения:**
- `key`: ровно 32 байта
- `nonce`: ровно 12 байт
- `aad`: дополнительные аутентифицированные данные (всегда пустое `b""` в GoyGram)

GCM обеспечивает **аутентифицированное шифрование** — если кто-то вмешается в файл хранилища, расшифровка завершится с ошибкой `"decryption failed (wrong key or corrupted data)"`.

### Служебные функции


```
cut(buf)  → (list_of_frames, remainder)
pack(data) → length_prefix + data
```


`cut` разбивает байтовый буфер на кадры с префиксом длины TL (4-байтовая длина с прямым порядком байтов + полезная нагрузка). Логика идентична версии Python в `MTNet.cut()`, но собственная скорость.

`pack` добавляет 4-байтовый префикс длины с прямым порядком байтов. Идентичен `IntermediateTransport.pack()`.

## Почему Rust, а не расширение Cython/C?

1. **Безопасность**: средство проверки заимствований Rust устраняет ошибки безопасности памяти. Криптовалютный код сильно манипулирует необработанными байтовыми буферами — неопределенное поведение здесь было бы катастрофой.
2. **Производительность**: ящик `aes` использует аппаратные инструкции AES-NI на платформе x86-64. Библиотека Python `cryptography` также делает то же самое, но накладные расходы на пересечение границы Python/C для каждого пакета складываются.
3. **Простота сборки**: `maturin` обеспечивает кросс-компиляцию для всех платформ. Нет необходимости поддерживать отдельные сценарии сборки расширений C для Linux/Win/Mac.
4. **Типовая безопасность**: PyO3 предоставляет нативные привязки Python к Rust. Сигнатуры функций (`&[u8]` → `PyResult<Vec<u8>>`) проверяются во время компиляции.

## Характеристики производительности

Путь шифрования/дешифрования IGE циклически обрабатывает 16-байтовые блоки. Для типичного пакета MTProto (около 500 байт, дополненных до 512, т. е. 32 блоков) это 32 вызова шифрования AES. При использовании AES-NI каждое шифрование занимает около 10–20 циклов ЦП. Полное шифрование пакета измеряется **наносекундами**.

Путь GCM использует `Aes256Gcm` из крейта `aes-gcm`, который также использует аппаратное ускорение. Шифрование хранилища обычно обрабатывает несколько КБ JSON — также наносекунды.

Настоящим узким местом является **никогда не криптовалюта**. Это сетевой ввод-вывод и выполнение обработчика Python.
