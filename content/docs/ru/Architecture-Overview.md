# Обзор архитектуры

GoyGram построен на **архитектуре с разделенным мозгом**: высокоуровневом Python API, который кажется эргономичным и питоническим, поддерживаемым модулем расширения Rust, который обрабатывает каждую криптографическую операцию и сериализацию TL на собственной скорости.

## Пятислойный стек


```
┌─────────────────────────────────────────────┐
│             GoyGram (Public API)             │  ← User-facing class
├─────────────────────────────────────────────┤
│        AppCore (Internal Engine)             │  ← Config, hooks, FSM, routing
├──────────────────┬──────────────────────────┤
│   BotNet (HTTP)  │   MTNet (TCP/MTProto)    │  ← Transport layer
├──────────────────┴──────────────────────────┤
│          Bus → Disp (Event Pipeline)         │  ← asyncio.Queue + dispatcher
├─────────────────────────────────────────────┤
│  goygram.ext (Rust .so) — AES/Crypto + TL   │  ← Native crypto & TL codec
└─────────────────────────────────────────────┘
```


### Уровень 1: общедоступный API (класс `GoyGram`)

Класс `GoyGram` в `client.py` представляет собой **фасад** — он делегирует все `AppCore`, но предоставляет чистую поверхность:

- Декораторы обработчиков: `on_msg`, `on_cb`, `on_cmd`, `on_poll`, `on_member`, `on_update`.
- Динамическая отправка методов: каждый метод Bot API и MTProto через `__getattr__`.
- Управление состоянием FSM: `set_state`, `get_state`, `get_state_data`, `clear_state`.
- Помощники форматирования: `html()`, `md()`.
- Помощники по транспортировке: `bot_req()`, `mt_req()`, `raw_chat()`, `via()`.
– Жизненный цикл: `run()`, `stop()`.
- Самоанализ: `help()`


```python
def __getattr__(self, name: str) -> Any:
    return getattr(self.core, name)
```


### Уровень 2: AppCore (внутренний движок)

`AppCore` владеет всей средой выполнения:

- **`bus`** — `asyncio.Queue` (`core/bus.py`) получение нормализованных событий событий от всех транспортов
- **`disp`** — класс `Disp` (`core/disp.py`), потребляющий события и перенаправляющий их обработчикам.
- **`fsm`** — `FSMEngine` (`core/fsm.py`) для конечных автоматов для каждого чата и пользователя.
- **`bot`** / **`mt`** — дополнительные экземпляры транспорта.
- **`api`** — необязательный экземпляр `BotAPI` (оболочка API-интерфейса типизированного бота).
- **Списки перехватчиков**: `hook` (сообщение), `cb_hook` (обратный вызов), `poll_hook` (опрос), `member_hook` (участник), `update_hook` (всеобъемлющий)

#### Разрешение динамического метода

Все методы Bot API и MTProto разрешаются через `__getattr__`:


```python
def __getattr__(self, name: str) -> Any:
    # Priority 1: BotAPI typed methods
    if self.api is not None and hasattr(self.api, name):
        return getattr(self.api, name)
    # Priority 2: MTProto methods (mt_ prefix)
    if name.startswith("mt_") and self.mt is not None:
        return self._dynamic_method(name)
    # Priority 3: Bot API methods (anything else)
    if not name.startswith("mt_") and not name.startswith("_") and self.bot is not None:
        return self._dynamic_method(name)
    raise AttributeError(name)
```


Snake_case автоматически преобразуется в CamelCase для Bot API и в `namespace.methodName` для MTProto. Никаких статических списков методов — все динамично.

### Уровень 3: Транспортный уровень

Два независимых сетевых транспорта, передающих один и тот же `Bus`:

| Транспорт | Класс | Протокол | Авторизация | Механизм обновления |
|-----------|-------|----------|------|-----------------|
| API ботов | `BotNet` (`vendor/botapi.py`) | HTTPS через aiohttp | Токен бота | Длинный опрос `getUpdates` |
| МТПрото | `MTNet` (`vendor/mtproto.py`) | Необработанный TCP-сокет | API ID/Hash + обмен ключами DH | Постоянное соединение, читает зашифрованные пакеты |

Оба нормализуют свои события в один и тот же формат dict и отправляют в `Bus`. Диспетчеру все равно, какой транспорт произвел событие.

### Уровень 4: Конвейер событий (шина + отображение)

**Bus** — это тонкая оболочка `asyncio.Queue`. События — это дикты `{"src": "bot"|"mt", "data": {...}}`.

**Диспетчер** запускает `consume()` в асинхронной задаче, вызывая `one()` для каждого события. `one()` соответствует шаблону `data["kind"]` (`"msg"`, `"cb"`, `"poll"`, `"member"`), создает соответствующий типизированный объект события и выполняет итерацию по зарегистрированным обработчикам. `StopPropagation` из любого обработчика останавливает текущую группу обработчиков.

### Уровень 5: Расширение Rust (`goygram.ext`)

Собственная библиотека Rust (скомпилированная с помощью `maturin` + PyO3, `opt-level=3`, `lto=true`, `strip=true`) предоставляет 12 функций:

| Функция | Цель |
|----------|---------|
| `aes_ige_enc` | Шифрование AES-256-IGE с дополнением PKCS7 |
| `aes_ige_dec` | Расшифровка AES-256-IGE + unpad |
| `aes_ige_enc_raw` | Шифрование AES-256-IGE без заполнения |
| `aes_ige_dec_raw` | Расшифровка AES-256-IGE без заполнения |
| `aes_gcm_encrypt` | Шифрование с аутентификацией AES-256-GCM |
| `aes_gcm_decrypt` | Расшифровка с аутентификацией AES-256-GCM |
| `cut` | Разделитель кадров для протоколов с префиксом длины |
| `pack` | Пакер с префиксом длины |
| `load_schema` | Загрузить разобранную JSON-схему TL в память |
| `serialize_method` | Динамическая сериализация вызова метода TL |
| `tl_method_exists` | Проверьте, известен ли метод TL |
| `schema_loaded` | Проверьте, загружена ли схема TL |

Функции IGE являются критическим путем для MTProto — каждый зашифрованный пакет проходит через AES-IGE. GCM используется для шифрования хранилища. Среда выполнения схемы TL в Rust обрабатывает динамическую сериализацию любого метода MTProto из проанализированной схемы `.tl`, избегая генерации кода во время выполнения.

### Двигатель ФСМ

`FSMEngine` (`core/fsm.py`) обеспечивает управление состоянием каждого чата и пользователя:

- Под ключом кортежа `(chat_id, user_id)`
- Срок действия на основе TTL (по умолчанию 3600 с), фоновая очистка каждые 600 с.
- `set()` создает или обновляет состояние с помощью дополнительного словаря данных — существующие данные **объединяются** при обновлении.
- `get()` возвращает название штата или `None`; `get_data()` возвращает копию данных или `None`
- `clear()` удаляет состояние
- Фильтры: `state("name")`, `state_any("a", "b")` для маршрутизации обработчика.

### Почему эта архитектура

1. **Python для эргономики**: обработчики на основе Decorator, конфигурации pydantic, async/await, расширенный TUI. Оркестровка в Rust была бы усложнением без пользы.

2. **Ржавчина ради скорости**: AES-256-IGE в Python будет на несколько порядков медленнее. Сериализация TL сложных запросов MTProto также выигрывает от собственной скорости. Оба являются критическими путями для каждого сообщения.

3. **Чистая граница**: модуль Rust представляет собой чисто криптографическую библиотеку кодеков TL. Нулевые знания Telegram, сети MTProto или HTTP. Поменяйте местами серверную часть, и ничего выше не изменится.

4. **Шина одного события**: оба транспорта помещаются в одну очередь. Диспетчеру плевать на происхождение. Именно это делает двойной транспорт безупречным.

5. **Динамическая отправка**: нет жестко закодированных оболочек методов. `__getattr__` делает доступными все API-интерфейсы ботов и методы MTProto без необходимости создания кода или обслуживания.
