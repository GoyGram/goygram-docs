# Многосессионная архитектура

GoyGram поддерживает запуск нескольких независимых сеансов MTProto из одной и той же базы кода, каждый со своим собственным файлом хранилища, аутентификацией и подключением. Никакого совместного использования сеансов, никакого общего состояния — чистая изоляция.

## Именованные сеансы


```python
app1 = GoyGram(
    api_id=API_ID, api_hash=API_HASH,
    session_name="account_1"
)

app2 = GoyGram(
    api_id=API_ID, api_hash=API_HASH,
    session_name="account_2"
)
```


Каждый сеанс получает свой собственный файл хранилища:
- `account_1.vault` (зашифровано)
- `account_2.vault` (зашифровано)

## Как работает изоляция

### Отдельные файлы

Параметр `session_name` управляет именем файла хранилища:


```python
vault = Path(f"{session_name}.vault")
```


Изменение `session_name` меняет файл — вот и весь механизм изоляции. Каждый экземпляр читает/записывает другое хранилище.

### Отдельный вывод ключа

Ключ хранилища включает `session_name` в материале деривации:


```python
material = f"{_get_machine_id()}:{session_name}".encode()
key = hashlib.pbkdf2_hmac("sha256", material, salt, 600000, dklen=32)
```


Разные имена сеансов → разные ключи → разные зашифрованные объекты. Даже если два хранилища находятся на одной машине, знание одного ключа не поможет при работе с другим.

### Отдельные соединения

Каждый экземпляр `GoyGram` создает свой собственный сокет `AppCore` → `MTNet` → TCP. Нет совместного использования ресурсов:
- Отдельные TCP-соединения
- Отдельный обмен ключами DH
- Отдельные ключи авторизации
- Отдельные порядковые номера
- Отдельные идентификаторы сообщений

### Отдельные циклы событий

Если вы запускаете их в одном процессе, они используют общий цикл событий, но имеют отдельные асинхронные задачи:


```python
import asyncio

async def main():
    app1 = GoyGram(api_id=..., api_hash=..., session_name="acc1")
    app2 = GoyGram(api_id=..., api_hash=..., session_name="acc2")

    @app1.on_msg(filt=filters.text)
    async def handler1(msg): ...

    @app2.on_msg(filt=filters.text)
    async def handler2(msg): ...

    await asyncio.gather(app1.run(), app2.run())
```


### Отдельный self_id

Каждый экземпляр отслеживает свой собственный идентификатор пользователя:


```python
app.self_id = uid
app.mt.self_id = uid
```


Фильтр `filters.me` использует `app.self_id`, чтобы определить, пришло ли сообщение «от меня», поэтому `filters.me` правильно идентифицирует сообщения от нужной учетной записи в каждом экземпляре.

## Практические шаблоны многосессионной работы

### Пул фермерских рабочих


```python
workers = []
for i in range(5):
    app = GoyGram(
        api_id=API_ID, api_hash=API_HASH,
        session_name=f"worker_{i}"
    )

    @app.on_cmd(".ping")
    async def ping_handler(msg):
        await msg.reply(f"Worker {i} alive")

    workers.append(app)

await asyncio.gather(*[w.run() for w in workers])
```


### Комбинация «Бот + Пользователь»


```python
bot = GoyGram(bot_token="123:ABC")
user = GoyGram(api_id=123, api_hash="abc", session_name="user")

await asyncio.gather(bot.run(), user.run())
```


## Ограничения

- **Одинаковые учетные данные API**: все сеансы используют один и тот же `api_id`/`api_hash` в конструкторе. В хранилище они хранятся для каждого сеанса, но вы не можете использовать разные учетные данные для разных сеансов без отдельных вызовов конструктора `GoyGram`.
- **Без объединения сеансов**: каждый сеанс представляет собой независимый экземпляр `GoyGram`. Нет встроенного пула сеансов или фабрики.
- **Память процесса**: каждый сеанс потребляет ~10–50 МБ памяти (объекты Python + расширение Rust + буферы TCP). Примерно через 50 сеансов вы это почувствуете.
- **Нет межсессионного состояния**: сеансы не могут напрямую обращаться к обработчикам или состоянию друг друга. Если вам нужна координация, используйте внешнюю очередь или общую переменную состояния (с соответствующей блокировкой).
