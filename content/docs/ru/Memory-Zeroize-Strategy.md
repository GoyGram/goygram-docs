# Стратегия обнуления памяти

GoyGram агрессивно очищает конфиденциальные данные из памяти с помощью своей стратегии «Пожиратель сеансов» (обнуление). Это относится в первую очередь к миграции сеансов и работе с хранилищем.

## Что обнуляется

### 1. Устаревшие файлы `.session`

При переходе с файла Telethon/Pyrogram `.session` на зашифрованный файл `.vault`:


```python
def _zeroize_and_remove(path: Path) -> None:
    size = path.stat().st_size
    with path.open("r+b") as f:
        f.write(b"\x00" * size)     # overwrite every byte with 0x00
        f.flush()
        os.fsync(f.fileno())        # force to physical disk
    path.unlink(missing_ok=True)    # delete the file
```


При этом весь файл перезаписывается нулевыми байтами, синхронизируется с диском, а затем удаляется. Вызов `os.fsync` гарантирует, что перезапись достигнет устройства хранения до того, как файл будет удален.

### 2. Повторное шифрование хранилища

Когда обнаруживается простое хранилище JSON (`raw[0] == 0x7B`), оно считывается, а затем **повторно шифруется** при следующей записи:


```python
if raw[0] == 0x7B:
    log.info("Vault %s is in plain JSON format, will re-encrypt on next save.")
    return json.loads(raw.decode())
```


Данные JSON в виде открытого текста существуют в памяти только во время окна чтения и повторного шифрования. После вызова `_write_vault` он шифруется с помощью AES-256-GCM, и открытый текст подлежит сборке мусора.

## Что НЕ обнуляется (известные ограничения)

### Объекты Python

Python не поддерживает явное обнуление памяти. Когда `auth_key`, `user` или другие конфиденциальные словари выходят за пределы области действия, они подлежат сборке мусора, но CPython не обнуляет память перед ее освобождением. Данные могут сохраняться в куче процесса до тех пор, пока не будут перезаписаны новыми выделениями.

### Буферы расширения Rust

Код Rust выделяет `Vec<u8>` для шифрования/дешифрования. Rust удаляет их, когда они выходят за пределы области видимости, но не обнуляет их. Ящик `secrets` или ящик `zeroize` можно использовать, но в настоящее время он не используется.

### Переменные стека

Локальные для функции переменные, содержащие ключевой материал (производные ключи, IV), живут в стеке и не очищаются явно. Когда функция возвращает значение, пространство стека используется повторно, но не обнуляется.

## Практическая безопасность

Стратегия обнуления нацелена на наиболее уязвимую поверхность: **файлы на диске**. Файл `.session`, находящийся в вашем домашнем каталоге, гораздо более уязвим для атаки, чем память процесса. Обнуление и удаление гарантирует, что даже если кто-то получит доступ к файловой системе после миграции, старый файл сеанса будет невозможно восстановить.

Для глубокоэшелонированной защиты:


```bash
# Use encrypted filesystem
# /home mounted on LUKS/dm-crypt

# Restrict vault file permissions
chmod 600 *.vault

# Environment variable override for headless servers
export GOYGRAM_VAULT_KEY=$(cat /run/secrets/goygram_key)
```


Шифрование хранилища AES-256-GCM защищает хранящиеся данные независимо от обнуления памяти.
